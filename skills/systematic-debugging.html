Systematic Debugging

재현 우선, 근본 원인 우선, 실패 테스트 우선. "하면서 다른 것도 고치는" 행위를 차단하는 엄격한 디버깅 워크플로입니다.

▸ 언제 사용하나요?

버그, 테스트 실패, 예상치 못한 동작 — 무엇이든 고쳐야 할 때.
  • 런타임 에러, 테스트 실패, 예기치 않은 동작
  • Run Plan 또는 Long Run 실패 시 자동 전환
  • "이 테스트가 불안정해", "버그 있어"

▸ 7단계 디버깅 워크플로

핵심 원리: 재현 없으면 수정 없습니다. 가설 없으면 수정 없습니다. 한 번에 하나의 가설만.
1
문제 정의 (한 문장)

"무엇이 기대와 다른가"를 한 문장으로 명확히 합니다.

2
재현 또는 계측

최소 재현 경로를 확보합니다. 재현 불가면 계측부터.

3
컴포넌트 경계에서 증거 수집

입력/출력을 추적하여 실패 지점을 좁힙니다.

4
단일 근본 원인 가설 분리

"A가 B 때문에 실패한다" — 하나의 구체적 가설.

5
실패 테스트로 고정

가설을 증명하는 실패 테스트를 작성합니다.

6
가설당 하나의 수정

한 가설, 한 수정. 리팩토링 금지.

7
검증 및 종료

테스트 통과 → 종료. 3회 실패 시 구조적 문제 분석.

⛔ HARD GATE: 재현 없이 수정 금지. 가설 없이 수정 금지. 동시에 여러 가설 금지.

▸ 하드 게이트 규칙

규칙설명
재현 금지재현 없이는 수정하지 않음
가설 금지가설 없이는 수정하지 않음
한 번에 하나동시 다중 가설/수정 금지
"하면서" 금지"while I'm here" 리팩토링 금지

▸ 참고 가이드

  • 조건 기반 대기 — 불안정한 타임아웃을 안정적인 조건 폴링으로 교체
  • 심층 방어 검증 — 데이터가 통과하는 모든 레이어에서 검증
  • 근본 원인 추적 — 호출 체인을 역추적하여 원래 트리거 찾기
  • 오염자 찾기 스크립트 — 원치 않는 파일/상태를 생성하는 테스트를 찾는 이분법 스크립트
