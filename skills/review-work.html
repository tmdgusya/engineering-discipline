Review Work

정보가 격리된 실행 후 검증. 계획 문서와 코드베이스만 읽습니다 — 실행 로그나 worker 출력은 받지 않습니다.

언제 사용하나요?

Run Plan 또는 Long Run 완료 직후. "구현이 계획을 제대로 따랐는가?"를 독립적으로 검증해야 할 때.
  • "작업 검토해줘", "구현 확인해줘"
  • run-plan 완료 후 자동 전환

어떻게 동작하나요?

핵심 원리: Reviewer는 정보 격리됩니다. 계획 문서만 받고, 실행 로그나 worker 출력은 받지 않습니다. 구현자의 의도가 아니라 오직 결과만 평가합니다.
1
계획 로드 및 파싱

계획 파일만 읽습니다. 목표, 범위, 수락 기준을 추출.

2
코드베이스 검사

계획된 파일 vs 실제 파일을 대조. 잔여 아티팩트(TODO, 비어있는 함수 등) 탐지.

3
테스트 독립 실행

계획에 명시된 테스트를 직접 실행합니다 — worker의 테스트 결과를 신뢰하지 않음.

4
Git 히스토리 검증

계획된 커밋이 실제로 있는지 히스토리와 대조합니다.

5
이진 판정 (PASS/FAIL)

구조화된 검토 문서를 docs/engineering-discipline/reviews/에 저장합니다. 코드를 수정하지 않습니다 — 보고만 합니다.

왜 정보를 격리하나요?

Worker의 출력을 본 Reviewer는 "의도는 좋았으니 통과"하는 편향을 가집니다. 정보 격리는 결과만으로 검증의 공정성을 보장합니다.
