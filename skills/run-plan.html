03 실행

Run Plan

Worker가 구현하고 Validator가 독립 검증합니다. 실패 시 자동 재시도, 3회 실패 시 사용자에게 에스컬레이션합니다.

01 ▸ 워크플로우 위치

Plan Crafting (계획 완료) │ ▼ ┌─────────────────────────────┐ │ Run Plan (ACTIVE) │ │ Worker-Validator 루프 실행 │ └──────────────┬──────────────┘ │ ▼ Review Work (최종 독립 검증)

02 ▸ 언제 사용하나요?

Plan Crafting이 완료된 직후, 또는 이미 작성된 계획 문서를 실제 코드로 구현해야 할 때 사용합니다.
주요 트리거
  • 계획 실행해
  • 플랜 돌려줘
  • 작성된 계획대로 구현 시작해

03 ▸ 어떻게 동작하나요?

핵심 원리: Validator는 Worker의 코드를 절대 보지 않습니다. 오직 계획의 목표와 수락 기준만으로 결과물이 올바른지 독립적으로 검증합니다.
단계별 실행 프로세스
  1. 검증 인프라 발견: 현재 환경에서 사용 가능한 테스트 러너, 린터, 에이전트 스킬을 파악합니다.
  2. 계획 사전 검토: 구현 전 파일 충돌 가능성이나 의존성 순서 오류를 최종 확인합니다.
  3. Worker-Validator 루프: Worker가 작업을 수행하면 Validator가 독립적으로 결과물을 검토합니다.
  4. E2E 검증 게이트: 개별 단계가 모두 완료되면 전체 시스템의 통합 동작을 확인합니다.

04 ▸ 병렬 실행 규칙

조건실행 방식
파일 겹침 없음 + 의존성 없음병렬 실행 가능
동일 파일 수정 필요직렬 — 앞 작업 완료 후 수행
작업 A의 결과가 B의 입력인 경우직렬 — 순차적 의존성 준수

05 ▸ 하드 게이트

  • 3회 실패 제한: 동일 단계에서 3회 연속 검증 실패 시 즉시 중단하고 사용자에게 에스컬레이션합니다.
  • 정보 격리 준수: Validator는 Worker의 구현 로그를 보지 못하며, 오직 계획된 수락 기준만 대조합니다.
  • 검증 없는 완료 금지: 모든 코드 변경은 반드시 자동화된 테스트나 명확한 확인 절차를 통과해야 합니다.
