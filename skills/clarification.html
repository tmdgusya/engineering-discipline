Clarification

반복적인 Q&A + 병렬 코드베이스 탐색을 통해 모호한 요청을 명확한 작업 범위로 좁힙니다.

▸ 언제 사용하나요?

~하고 싶어, ~이 필요해, ~을 만들자 — 범위가 즉시 명확하지 않은 모든 요청의 출발점입니다.
  • 요청이 모호해서 구현 방향이 여러 갈래일 때
  • 기존 코드베이스와 충돌할 가능성이 있을 때
  • 사용자 자신이 정확히 무엇을 원하는지 아직 정리하지 못했을 때

▸ 어떻게 동작하나요?

핵심 원리: 사용자 질문 트랙 1, 코드 탐색 트랙 2 — 두 트랙이 동시에 진행되며 서로를 보완합니다.
1
모호성 파악

요청에서 무엇이 불명확한지 식별합니다.

2
질문 1개 + 코드 탐색 병렬 dispatched

메시지당 오직 하나의 질문만 합니다. 동시에 서브에이전트가 코드베이스를 탐색합니다.

3
답변 + 탐색 결과 합성

사용자의 답변과 코드 탐색 결과를 교차 검증합니다. 여전히 모호하면 다음 질문.

4
Context Brief 생성

목표, 범위, 기술 컨텍스트, 제약사항, 성공 기준, 복잡도 평가를 문서화합니다.

5
복잡도 기반 자동 라우팅

점수 5-8 → Plan Crafting / 점수 9-15 → Milestone Planning

위험 표면통합 위험 없음내부 통합외부 시스템
  • 메시지당 질문 단 1개 — 복합 질문 금지
  • 항상 서브에이전트로 코드베이스 탐색 — 독단에 의존하지 않음
  • 명확해질 때까지 구현 시작 금지 — "이제 충분하다"는 자체 판단 금지
  • 질문은 항상 범위를 좁혀야 함 — 같은 수준의 추상화 반복 금지
