01 출발점

Clarification

반복적인 Q&A + 병렬 코드베이스 탐색으로 모호한 요청을 명확한 작업 범위로 좁힙니다.

01 ▸ 워크플로우 위치

사용자 요청 (모호) │ ▼ ┌─────────────────────────────┐ │ clarification (ACTIVE) │ │ 모호성 해소 + 코드베이스 탐색 │ └──────────────┬──────────────┘ │ ┌────────┴────────┐ ▼ ▼ 단순 (5-8) 복잡 (9-15) │ │ ▼ ▼ Plan Crafting Milestone Planning

02 ▸ 언제 사용하나요?

~하고 싶어, ~이 필요해, ~을 만들자 — 범위가 즉시 명확하지 않은 모든 요청의 출발점입니다.
주요 트리거
  • 요청이 모호해서 구현 방향이 여러 갈래일 때
  • 기존 코드베이스와 충돌할 가능성이 있을 때
  • 사용자 자신이 무엇을 원하는지 아직 정의하지 못했을 때

03 ▸ 어떻게 동작하나요?

핵심 원리: 사용자 질문 트랙과 코드 탐색 트랙이 동시에 진행되며 서로를 보완합니다.
단계별 실행
  1. 모호성 파악: 요청에서 불명확한 지점을 식별합니다.
  2. 질문 1개 + 코드 탐색 병렬: 메시지당 오직 하나의 질문만 하며, 서브에이전트가 코드베이스를 탐색합니다.
  3. 답변 + 탐색 결과 합성: 답변과 탐색 결과를 교차 검증합니다. 여전히 모호하면 다음 질문을 던집니다.
  4. Context Brief 생성: 목표, 범위, 제약사항, 성공 기준, 복잡도 평가를 문서화합니다.
  5. 복잡도 기반 자동 라우팅: 점수(5-15)에 따라 다음 워크플로우를 결정합니다.

05 ▸ 하드 게이트

반드시 지켜야 할 원칙
  • 메시지당 질문 단 1개 — 복합 질문 절대 금지
  • 항상 서브에이전트로 코드베이스 탐색 — 독단적인 추측 배제
  • 명확해질 때까지 구현 시작 금지 — "충분하다"는 주관적 판단 금지
  • 질문은 항상 범위를 좁혀야 함 — 추상적인 질문 반복 금지
